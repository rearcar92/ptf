커머스 연동 경험에 PG 정산과 운영 UX 관점을 더한 서비스 기획자

호스팅 커머스 환경에서 외부 서비스/API 연동과 운영 안정성을 경험하고, PG사에서 정산·결제 운영·관리자 UX 관점을 확장한 인하우스 서비스 기획자 포트폴리오 초안입니다.

Positioning Statement

이전에는 커머스 서비스가 외부 플랫폼과 안정적으로 연결되는 일을 경험했고, 이후에는 결제 이후의 돈의 흐름과 운영 리스크를 배웠습니다. 현재는 두 경험을 연결해 더 넓은 시야로 운영 문제를 서비스 구조로 풀고자 합니다.

Submission Note

본 문서는 레이아웃 검토용 초안입니다. 실제 수치, 기간, 팀 구성, 산출물 링크는 프로젝트별 사실 확인 후 보강합니다.

Project 01 · External Payment

네이버페이 연동 프로젝트

외부 연동 정책 정리 운영 적용

커머스 서비스에 외부 결제수단을 연결하기 위해 요구사항, 정책, 화면/운영 흐름, 개발 협의 사항을 정리한 경험으로 구성합니다.

Problem외부 결제수단 도입 시 회원, 결제, 운영 흐름에서 확인해야 할 정책과 예외 케이스가 존재했습니다.
Approach연동 요건, 화면 흐름, 운영 처리 기준을 정리하고 관련 부서와 적용 범위를 조율했습니다.
Result확인 필요: 적용 범위, 처리 건수, 오픈 일정, 운영 문의 감소 여부 등으로 보강합니다.
My Role확인 필요: 요구사항 정의, 정책 정리, 화면 설계, QA, 운영 적용 중 직접 맡은 범위를 표시합니다.
Lesson: 외부 결제 연동은 기능 연결보다 정책, 예외, 운영 기준 정리가 중요하다는 관점으로 정리합니다.
Project 02 · Member Integration

카카오싱크 연동 프로젝트

가입 흐름 동의 항목 회원 정보

카카오 계정 기반의 가입/동의/회원정보 처리 흐름을 서비스에 맞게 연결한 경험으로 구성합니다. 사용자 경험과 운영 정책이 만나는 지점을 강조합니다.

Problem가입과 동의 과정에서 사용자가 이해해야 할 정보와 서비스가 받아야 할 데이터 기준이 필요했습니다.
Approach동의 항목, 회원정보 매핑, 화면 문구, 예외 흐름을 기준으로 요구사항을 정리했습니다.
Result확인 필요: 연동 범위, 가입 흐름 변화, 운영 문의, 적용 대상 쇼핑몰 수 등으로 보강합니다.
My Role확인 필요: 정책/화면/개발 협의/테스트 중 실제 담당 범위를 분리합니다.

Project 03 · Security & Operation

ISMS 대응과 운영 안정성 정비

보안 대응 정책/증적 운영 기준

보안 전문가처럼 포장하지 않고, 서비스 기획자로서 정책, 화면/기능 수정, 권한·로그·개인정보 처리 기준, 증적 대응에 기여한 범위를 정리합니다.

Problem보안 인증 대응을 위해 서비스 정책과 운영 기준을 명확히 정리해야 했습니다.
Approach관련 화면, 기능, 운영 절차, 증적을 기준으로 담당 범위를 정리했습니다.
Result확인 필요: 대응 범위, 산출물 수, 협업 부서, 일정 준수 여부로 보강합니다.
My Role확인 필요: 정책서/화면 수정/증적/운영 가이드 중 직접 수행 범위를 표시합니다.
Project 04 · PG Settlement

PG 정산 업무와 운영 리스크 이해

정산 대사 운영 UX

PG사에서 배운 정산 업무를 돈의 흐름, 정산 주기, 수수료, 취소/환불, 대사, 보류/차감, 관리자 처리 관점으로 구조화합니다.

Problem결제 이후 정산 과정에는 운영자가 이해해야 할 예외와 리스크가 많았습니다.
Approach정산 흐름과 반복 이슈를 학습하고, 운영자가 확인해야 할 기준을 분리해 이해했습니다.
Result확인 필요: 담당 업무 범위, 반복 처리 유형, 개선 제안 여부, 운영 화면 관점으로 보강합니다.
My Role확인 필요: 정산 확인, 이슈 처리, 정책 정리, 화면 개선 제안 등 실제 범위를 표시합니다.
